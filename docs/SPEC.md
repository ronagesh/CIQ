## Overview

Ally is a competitor content intelligence dashboard for Amazon sellers. It audits product listing content (titles, bullets, descriptions, images) against Amazon's style guidelines, benchmarks against same-category competitors, and surfaces ready-to-use corrected copy. The primary user is a brand manager or content strategist responsible for a portfolio of Amazon SKUs.

---

## User flows

### 1. Brand selection (`/`)
User selects their brand from a dropdown populated from `data/scores.json`. Only brands with at least one product in the dataset appear. Submitting navigates to `/brand/[brand]`.

### 2. Brand dashboard (`/brand/[brand]`)
Displays all SKUs for the selected brand as issue cards, sorted by severity (suppression risk first, then high → medium → low, then by avg search rank).

**Controls:**
- SKU dropdown — filter to a single product or show all
- Refresh button — re-fetches the feed (reads from scores.json cache; does not re-run scoring)

**Each issue card shows:**
- Product title (truncated, full title on hover), brand, product ID
- Primary issue type label and severity badge (High / Medium / Low)
- Suppression risk badge if applicable
- Four QuantBar badges (title, bullets, description, images) — each shows Better / On par / Worse vs competitor averages for that category

**QuantBar logic:**
- Ratio = `productValue / competitorAvgValue`
- ≥ 1.08 → Better (emerald)
- ≤ 0.92 → Worse (amber)
- Otherwise → On par (zinc)
- If no competitor average available → no badge shown

Clicking a card navigates to `/report/[id]`.

### 3. Report page (`/report/[id]`)
Full content audit for a single SKU. Streams analysis from `/api/report/[id]`.

**Top row (3 columns):**
1. **Product tile** — thumbnail, title (hover to expand), product ID, category node, avg search rank
2. **Competitor set** — up to 5 ranked competitors in the same category node. Each row shows brand name and avg search rank. Clicking a row navigates to `/report/[competitor_id]`
3. **vs. Competitors scorecard** — Claude-assessed Better / On par / Worse per dimension (title, bullets, description, images), populated from the `scorecard` stream message

**Content Recommendations section:**
- Three highest-impact suggestions generated by Claude
- Accordion UI — click to expand each suggestion
- Each expanded suggestion shows:
  - Current text (read-only, red)
  - Proposed text (editable textarea, green, auto-sizes)
  - Real-time violation flags below the textarea (see Violation checking)
  - "Reset to original" link
  - Guideline citation (verbatim from style guide)
  - Competitor reference
  - Suppression risk badge if applicable

**Export:**
- "Generate Markdown Summary" button — always visible after streaming completes
- Label changes to "Regenerate Markdown" once markdown has been generated
- Editing any textarea clears existing markdown immediately
- Generated markdown uses the user's exact edited text verbatim — no LLM alteration

**Image Compliance section:**
- Audits main image of the product and top 3 competitors
- Shows per-image findings with pass/fail per check
- Suppression risk flagged inline
- Actionable fix suggestion if non-compliant
- Competitor images show a one-sentence comparison note

## Scoring logic (`scripts/seed.ts`)

Scores are computed offline and cached in `data/scores.json`. The brand dashboard reads from this cache — no LLM calls at browse time.

### Hard rule checks (deterministic, no LLM)

Applied in priority order. First violation found wins.

| Check | Severity | Suppression risk | Condition |
|-------|----------|-----------------|-----------|
| Title too long | high | yes | `title.length > 50` |
| Prohibited chars in title | high | yes (if ® © ™) | `/[!*$?®©™]/` |
| No images | high | yes | `imageCount === 0` |
| No bullet points | high | no | `bulletCount === 0` |
| No description | high | no | `descriptionLength === 0` |
| Description too long | medium | no | `descriptionLength > 2000` |

### LLM qualitative check

Only runs if no hard rule violations. Claude (Sonnet 4.6) is given the product content and the style guide and asked to identify the single most critical remaining qualitative issue, returning `severity`, `suppressionRisk`, `suppressionConsequence`, `issueType`, `rationale`.

Retry logic: up to 4 attempts with exponential backoff on 429/529.

### Competitor context

After scoring, the top competitor by avg search rank (if any) is appended to the rationale. `competitorAvgQuantitative` (averages across all same-category competitors) is stored alongside each issue card for use in QuantBar comparisons.

## Analysis pipeline (`/api/report/[id]`)

1. Load product and competitors from `data/products.csv`
2. Stream Claude Sonnet 4.6 analysis (see prompt spec below)
3. Accumulate full response text
4. Parse `{ scorecard, suggestions }` from the JSON response
5. **Post-process titles**: any suggestion with `dimension === "title"` and `proposedText.length > 50` is passed to `shortenTitle()` — a focused Haiku call that returns a ≤50 char version. Falls back to word-boundary truncation if the model overshoots.
6. Stream `suggestions` and `scorecard` messages to the client

### Analysis prompt structure

**System:** "You are an expert Amazon listing content strategist. Use only the rules in the style guide below. Return only valid JSON." + full style guide

**User prompt includes:**
- Selected SKU: brand, title (with char count / max), bullets (with count / target), description (with char count / max), image count, avg search rank
- Up to 5 competitors sorted best-ranked first: brand, avg search rank, title, bullets preview, description preview
- If product outranks all competitors: note to frame suggestions as guideline compliance, not competitive gaps

**Required output:**
```json
{
  "scorecard": {
    "title": "better" | "on par" | "worse",
    "bullets": "better" | "on par" | "worse",
    "description": "better" | "on par" | "worse",
    "images": "better" | "on par" | "worse"
  },
  "suggestions": [
    {
      "dimension": "Title" | "Bullet N" | "Description",
      "currentText": "...",
      "proposedText": "...",
      "guidelineCitation": "...",
      "competitorReference": "...",
      "suppressionRisk": true | false,
      "suppressionConsequence": "..."
    }
  ]
}
```

**Prompt rules enforced:**
- Exactly 3 suggestions
- Scorecard judges content quality, not just length
- Only cite rules that appear verbatim in the style guide
- Proposed title ≤50 characters (prompt instruction; also enforced in post-processing)
- Proposed description ≤2000 characters
- Prioritise suppression-risk issues first

## Image compliance logic (`/api/images/[id]`)

Analyzes the main image of the product + top 3 competitors using Claude Sonnet 4.6 vision. Each image is fetched and base64-encoded before being sent.

**Checks per image:**
1. White background (no colored backgrounds, lifestyle shots)
2. Product coverage ≥80% of image area
3. No text/watermarks/borders added as a digital layer
4. No promotional text overlaid on the photo
5. Product clearly visible and identifiable

**Critical overlay vs packaging distinction:**

A true digital overlay: semi-transparent banner, floating badge that doesn't follow the product's surface contour, sticker-style callout with drop shadow, text floating in the white space around the product.

Packaging surface text: words/logos printed on a box, label, pouch, or canister — they follow the surface curvature and share the same lighting/shadow as the product. These are never violations.

**Tie-breaker:** when in doubt, do NOT flag.

## Client-side violation checking

Real-time checks fire on every keystroke in the proposed text textarea. No API calls — pure regex against the style guide.

### Title checks
- `text.length > 50` → "{N} chars — max 50 (suppression risk)"
- `/[!*$?®©™]/` → "Prohibited characters: {chars}"
- `/\b[A-Z]{4,}\b/` → "Do not use ALL CAPS" (4+ char words only, avoids brand acronyms)
- `/\bfree\s+ship|\bsale\b/i` or `/\$\d/` → "No price, quantity, or promotional messages"
- `/\bbest\s+seller\b|\bhot\s+item\b/i` → "No subjective commentary"

### Bullet checks
- Ends with `.` or `!` → "No ending punctuation"
- Contains `!` → "No exclamation points"
- First character lowercase → "Must begin with a capital letter"
- `/\bfree\s+ship|\bsale\b/i` → "No promotional or pricing information"

### Description checks
- `text.length > 2000` → "{N} chars — max 2000"
- `/[A-Z]{5,}/` → "Do not write in ALL CAPS"
- `/@\w+|\bhttp/i` → "No email addresses or URLs"
- `/\bfree\s+ship|\bsale\b/i` → "No promotional language"

Editing any field immediately clears the markdown export panel.

## Markdown export

`generateSummary(product, suggestions)` is a pure synchronous function — no LLM involved. It builds the export from:
- The user's exact edited `proposedText` for each suggestion (verbatim, not re-processed)
- Original product text for dimensions that have no suggestion

Output format:
```markdown
# Content Update: {productId}

## Final Content
**Title**
{title}

**Bullet Points**
1. {bullet 1}
...

**Description**
{description}

## Change Log
| Field | Change | Guideline Cited | Competitor Referenced |
...
```

## Dataset

`data/products.csv` — 67 unique products, hydration/electrolyte category on Amazon.

Fields: `product_id`, `title`, `universe`, `image_url`, `bullet_points`, `min_rank_search`, `avg_rank_search`, `min_rank_category`, `avg_rank_category`, `retailer_category_node`, `retailer_brand_name`, `description_filled`

### Enrichment history
- Raw dataset: 154 rows → deduplicated to 67 (richest row per `product_id` by field coverage score)
- 15 products had missing `retailer_brand_name` → inferred via Claude Haiku from title + bullets
- 1 product had missing `retailer_category_node` → inferred via Claude Haiku
- Manual normalisation: "Liquid IV" → "Liquid I.V.", "Ultima" → "Ultima Replenisher", etc.

## Style guide limits (Amazon)

| Field | Rule |
|-------|------|
| Title | ≤50 chars; Title Case; no `! * $ ? ® © ™`; no ALL CAPS; no price/promo/subjective copy |
| Bullet points | ≤5; sentence fragments; no ending punctuation; no hyphens/exclamation points; start with capital |
| Description | ≤2000 chars; complete sentences; no ALL CAPS; no URLs/emails/seller info; no promo language |
| Main image | White background; product ≥80% of frame; no digital text overlays or watermarks; JPEG, RGB |
| Search terms | ≤250 chars total (5 fields × 50); single words; no title repeats |

Suppression triggers: title character limit violation, prohibited chars in title, no images, image rule violations (can also cause account suspension).

## Edge cases

### Data quality
- **No competitors in category** — Amazon categories are hierarchical breadcrumbs, e.g. "Health & Household › Sports Nutrition › Electrolyte Replacements". If no other products in the dataset share a product's exact category, the system automatically widens the search one level up (e.g. to "Health & Household › Sports Nutrition") to find at least some competitors. If that broader bucket is also empty, the competitor tile and scorecard show empty states. The analysis prompt notes "No ranked competitors available" and Claude frames suggestions purely as guideline compliance.
- **Competitor with no search rank** — unranked competitors sort to the bottom of the competitor set. The scorecard still uses them for qualitative assessment but the competitor reference in suggestions avoids citing an unranked SKU as a benchmark.
- **Product outranks all competitors** — the analysis prompt injects a note instructing Claude to frame suggestions as suppression-risk and compliance issues rather than competitive gaps, so suggestions don't read as "improve vs a worse competitor."
- **Missing brand name** — the current dataset has been fully enriched so all 67 products have a brand name. However, if new products are added to the CSV without first running the enrichment script, any product with an empty `retailer_brand_name` will display as "(unknown)" and be invisible from the home screen — the brand dropdown only shows brands with non-empty names. Running `npx tsx scripts/enrich.ts` before re-seeding prevents this.
- **Malformed CSV fields** — `image_url` and `bullet_points` are JSON arrays serialised as strings. `parseImages` and `parseBullets` fall back to treating the raw string as a single-element array if `JSON.parse` fails, so malformed rows degrade gracefully rather than crashing the loader.
- **Image fetch timeout** — `fetchImageAsBase64` uses an 8-second `AbortSignal.timeout`. On timeout or non-200 response it returns `null` and that product is skipped from the image analysis results. The UI shows "No images available to analyze" if all fetches fail.
- **Streaming parse failure** — the stream accumulates raw text and attempts one JSON parse at the end. If the model returns malformed JSON (truncated response, extra commentary outside the JSON block), the `try/catch` around the parse silently swallows the error and no `suggestions` or `scorecard` message is sent. The UI stays in the skeleton/empty state rather than crashing.
- **Title shortener overshoot** — if the Haiku `shortenTitle` call still returns > 50 chars (rare), a word-boundary truncation regex clips it to the last whole word at or under 50 chars as a final safety net.
- **LLM rate limits during seeding** — `scoreProduct` retries up to 4 times with exponential backoff (2s, 4s, 8s) on HTTP 429/529. Products that exhaust retries are logged as failures but don't abort the whole seed run; the rest of the batch still completes.

### UI edge cases
- **Very long titles in cards** — titles are clamped to 2 lines with a `title` tooltip for the full text. In the report header they're truncated with a CSS `truncate` class.
- **Accordion state across navigation** — `expandedIdx` is local component state and resets on every report page load. There is no persistence of which accordion item was open.
- **Regenerate after edit** — editing any textarea clears `markdown` state immediately. If the user then navigates away and back, the markdown is gone (no persistence). They must click Generate again.
- **Competing violation flags** — if both "No ending punctuation" and "No exclamation points" are triggered by the same `!`, both flags render. This is intentional — each cites a separate guideline.

## Future optimisations (if launching to production)

### Authentication and multi-tenancy
The current build has no authentication. In production, brand selection would be gated behind a login so each brand manager only sees their own SKUs. The `/brand/[brand]` URL pattern would need to be protected — currently any user can view any brand by editing the URL.

### Real-time data instead of a seeded CSV
`data/scores.json` is a static cache written by a manual script. In production this would be replaced by:
- A database (e.g. Postgres) storing products and scores
- A background job that re-scores products on a schedule or when new data arrives
- An API integration pulling live listing data directly from Amazon Selling Partner API (SP-API) instead of a CSV export

### Incremental re-scoring
The seed script re-scores every product on every run. With 67 products this takes ~2 minutes. At scale, a smarter approach would hash each product's content fields and only re-score rows where the hash has changed since the last run.

### Caching image analysis
Image compliance analysis is expensive — one vision API call per image, run fresh on every report page load. In production, results should be cached (keyed by image URL + a content hash) and invalidated only when the image changes. A simple key-value store (e.g. Vercel KV or Redis) would suffice.

### Bulk export
Currently the markdown export covers one SKU at a time. Brand managers typically want to export a full remediation brief for all high-severity SKUs in one go — a CSV or ZIP of markdown files, one per product.

### Historical tracking
There is no record of what a listing looked like before vs after Ally's suggestions were applied. In production, storing a snapshot of the listing content at audit time would allow a "before/after" view and let teams measure whether content scores improved after applying recommendations.

### Confidence scoring on image compliance
The image audit currently gives a binary pass/fail per check. A confidence score (high / medium / low) would reduce noise — a borderline background that is "mostly white but slightly off" is not the same risk as a product that is clearly on a lifestyle background.

### Suggestion feedback loop
There is no way for users to signal that a suggestion is wrong or unhelpful. A thumbs-down on a suggestion, fed back into a fine-tuning dataset or a system-prompt improvement loop, would improve quality over time.

### Category breadth
The current dataset covers one category (hydration/electrolytes). The style guide and scoring rules are written to be category-agnostic, but the competitor comparisons and Claude's framing of suggestions would be more accurate with category-specific context injected into the prompt (e.g. average title length norms differ significantly between Electronics and Grocery).

### Search term audit
The style guide includes rules for backend search terms (≤250 chars, no title repeats, single words) but Ally does not audit them. Search terms are not exposed in the current CSV export and would require SP-API access.

### A+ content
Amazon A+ content (enhanced brand content) has its own guidelines and significantly impacts conversion. Auditing A+ modules — image/text layout, headline length, comparison charts — is a natural extension of the current content audit scope.

### Marketplace localisation
The current rules are for Amazon US. UK, DE, FR, JP, and other marketplaces have different character limits (some use 80-char title limits), different prohibited character sets, and different suppression thresholds. Supporting multiple marketplaces would require a per-marketplace style guide layer.
